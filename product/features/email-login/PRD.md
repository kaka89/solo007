# 邮箱登录功能需求文档

## 1. 概述

### 1.1 背景
当前产品仅支持手机号验证码登录，但部分用户更习惯使用邮箱进行注册和登录。为扩大用户覆盖、降低注册门槛，需要增加邮箱登录功能。

### 1.2 目标
- 允许用户使用邮箱地址进行注册和登录
- 提供与手机号登录一致的用户体验
- 支持邮箱验证流程，保障账户安全
- 为后续邮箱绑定、找回密码等功能奠定基础

### 1.3 范围
- 前端：邮箱输入、验证码发送、登录/注册界面
- 后端：邮箱验证、验证码生成与校验、用户账户创建/验证
- 非功能：安全性、性能、可扩展性

## 2. 目标用户

| 用户类型 | 使用场景 | 需求痛点 |
|----------|----------|----------|
| 新用户 | 首次注册 | 不想透露手机号，或习惯使用邮箱注册 |
| 老用户 | 登录现有账户 | 手机号更换、信号不佳、国际漫游 |
| 企业用户 | 团队注册 | 使用公司邮箱统一管理 |
| 海外用户 | 国际服务 | 手机号格式不支持、无本地号码 |

## 3. 功能需求

### 3.1 核心流程
1. **输入邮箱**：用户在登录页选择“邮箱登录”，输入邮箱地址
2. **发送验证码**：系统向该邮箱发送6位数字验证码（有效期5分钟）
3. **输入验证码**：用户收到邮件后，在页面输入验证码
4. **登录/注册**：
   - 若邮箱已注册：直接登录进入系统
   - 若邮箱未注册：自动创建账户并登录（需补全昵称等基本信息）

### 3.2 功能点清单

#### 3.2.1 前端界面
- 登录页增加“邮箱登录”切换入口
- 邮箱输入框（带格式校验）
- 验证码输入框（6位数字）
- “发送验证码”按钮（防刷策略：60秒冷却）
- 邮箱未注册时的昵称补全表单（可选）

#### 3.2.2 后端服务
- 邮箱格式校验（RFC 5322标准）
- 验证码生成与存储（Redis，5分钟过期）
- 邮件发送服务集成（SMTP/第三方服务）
- 用户账户查询与创建
- 登录态签发（JWT）

#### 3.2.3 安全要求
- 验证码防刷：同一邮箱每分钟最多1次，每日最多10次
- 验证码防猜：错误尝试超过5次锁定15分钟
- 邮箱黑名单：禁止临时邮箱、已知垃圾邮箱域名
- 登录态安全：HTTPS、Token刷新机制

## 4. 用户故事

### 4.1 新用户注册
> 作为新用户，我希望使用我的工作邮箱注册账户，以便使用产品服务，同时避免提供手机号。

**验收条件**：
- 输入合法邮箱后，能收到验证码邮件
- 输入正确验证码后，进入昵称补全页面
- 补全昵称后，成功进入首页，并显示欢迎提示

### 4.2 老用户登录
> 作为已注册用户，我希望用邮箱直接登录，避免每次都用手机验证码。

**验收条件**：
- 输入已注册邮箱，发送验证码
- 输入正确验证码后，直接进入首页
- 用户信息、设置、历史数据均正常加载

### 4.3 验证码未收到
> 作为用户，如果没收到验证码邮件，我希望能重新发送，并了解可能的原因。

**验收条件**：
- 发送按钮60秒后可再次点击
- 提供“未收到邮件？”帮助链接，列出常见原因（垃圾邮件、邮箱错误等）
- 支持更换邮箱地址

## 5. 验收标准

### 5.1 功能验收
- [ ] 邮箱格式校验：正确格式通过，错误格式提示
- [ ] 验证码发送：成功发送返回提示，失败显示原因
- [ ] 验证码验证：正确验证码登录成功，错误验证码提示
- [ ] 自动注册：未注册邮箱验证后进入补全信息流程
- [ ] 会话保持：登录后刷新页面仍保持登录状态

### 5.2 安全验收
- [ ] 验证码防刷：频繁请求被限制
- [ ] 验证码防猜：多次错误尝试被锁定
- [ ] 邮件内容不含敏感信息（仅验证码）
- [ ] 登录态不可在非HTTPS环境下传输

### 5.3 性能验收
- [ ] 验证码发送平均延迟 < 2秒
- [ ] 登录验证平均延迟 < 500毫秒
- [ ] 支持并发1000次/分钟的验证码发送

## 6. 非功能需求

### 6.1 可用性
- 邮件送达率 > 99%
- 验证码邮件进入收件箱（非垃圾邮件）概率 > 95%
- 界面响应时间 < 1秒

### 6.2 兼容性
- 支持主流邮箱服务商（Gmail、Outlook、QQ、163、企业邮箱等）
- 前端兼容 Chrome 90+、Safari 14+、Edge 90+

### 6.3 可维护性
- 邮件模板可配置（支持变量替换）
- 验证码策略（长度、有效期）可配置
- 黑名单域名可动态更新

## 7. 依赖与约束

### 7.1 依赖项
- 邮件发送服务（已有或需集成）
- Redis缓存服务（存储验证码）
- 用户数据库（存储邮箱与用户关系）

### 7.2 技术约束
- 后端API需保持与手机登录接口一致的结构
- 前端组件需复用现有登录页样式
- 不支持邮箱密码登录（仅验证码）

### 7.3 业务约束
- 同一邮箱只能绑定一个账户
- 已通过手机注册的用户，可后续绑定邮箱
- 邮箱验证后，需通过手机或邮箱二次验证方可修改敏感信息

## 8. 后续迭代

### 8.1 短期优化（v1.1）
- 邮箱绑定功能（手机用户添加邮箱）
- 找回密码功能（通过邮箱重置）
- 邮箱验证状态展示

### 8.2 长期规划
- 支持邮箱+密码登录
- 支持第三方邮箱OAuth（Google、Microsoft）
- 邮箱别名支持

## 9. 附录

### 9.1 邮箱格式校验规则
- 符合RFC 5322标准
- 本地部分：允许字母、数字、特殊字符 .!#$%&'*+/=?^_`{|}~-
- 域名部分：有效域名，至少一个点
- 长度限制：本地部分≤64字符，域名部分≤255字符，总长≤320字符

### 9.2 验证码规则
- 6位数字，随机生成
- 有效期5分钟
- 大小写不敏感
- 使用后立即失效
